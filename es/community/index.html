Proyectos y comunidades existentes

Si contribuyes a un proyecto existente o lo amplías, casi siempre es más fácil seguir usando la licencia de ese proyecto. Para encontrarla, busca un archivo llamado LICENSE o COPYING, y echa un vistazo al README del proyecto. Si no encuentras una licencia, pregunta a los mantenedores.

Según la licencia del proyecto original, usar la misma licencia puede ser un requisito, y no solo lo más fácil. (Consulta la condición «misma licencia» de algunas licencias.)

Algunas comunidades tienen fuertes preferencias por licencias concretas. Si quieres participar en alguna de ellas, será más fácil usar su licencia preferida, aunque empieces un proyecto totalmente nuevo sin dependencias existentes. Algunos ejemplos:

  • Apache exige la Apache License 2.0
  • La Cloud Native Computing Foundation exige la Apache License 2.0
  • GNU recomienda la GNU GPLv3 para la mayoría de los programas
  • Los paquetes de npm usan de forma abrumadora las licencias MIT o la muy similar ISC
  • OpenBSD prefiere la licencia ISC
  • Los crates de Rust se publican de forma abrumadora bajo MIT OR Apache License 2.0
  • Los plugins y temas de WordPress deben ser GNU GPLv2 (o posterior)
  • Las extensiones y plantillas de Joomla deben ser GNU GPLv2 para el código PHP.

Las comunidades son de todas las formas y tamaños, y más de una comunidad podría ser pertinente (por ejemplo, ten en cuenta a tu empresa si trabajas para una). Los ejemplos anteriores están muy asentados. Si la comunidad para la que construyes un proyecto no tiene tradiciones de licencia grabadas en piedra, o si no consideras que tu proyecto forme parte de ninguna comunidad en concreto, no pasa nada: haz tu propia elección de una licencia de código abierto.
